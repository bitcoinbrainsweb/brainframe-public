# Technical Writing
Version: 1.0

Produce documentation that is clear, accurate, and appropriate for its audience.

## Audience Types

- **End users** — plain language, action-oriented, no jargon
- **Technical collaborators** — concrete, example-driven, structured
- **Clients** — formal, professional, outcome-focused
- **Regulators / auditors** — precise, evidence-backed, no hedging

## Core Principles

1. **Audience first.** Identify who reads this before writing one word.
2. **Structure before prose.** Outline sections before writing sentences.
3. **Active voice.** "The system does X" not "X is done by the system."
4. **No hedging.** Say what is true. If uncertain, flag it explicitly.
5. **One idea per sentence.** No compound sentences hiding complexity.

## Review Checklist

- [ ] Would the target audience understand this without help?
- [ ] Is every claim verifiable or sourced?
- [ ] Are all acronyms defined on first use?
- [ ] Is the structure scannable (headers, short paragraphs)?
- [ ] Is there anything that could be misread by a hostile reader?
